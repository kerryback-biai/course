Two Separate Decisions

Before we get into the Morgan Stanley discussion, I want to plant one important idea that connects back to Session 3.

[TIMING: Brief transition slide. Move to the next slide immediately.]

Where Does It Run?

Where does YOUR CODE run?

  • Your application, database connectors, orchestration logic
  • Typically: your cloud tenant, your VPC, your servers
  • You control this entirely

Where does THE LLM run?

  • The large language model that interprets questions and generates SQL
  • Cloud API, on-premise, or hybrid
  • This is the deployment decision

Bridge from Session 3: the agent architecture has two halves. Your code and the LLM are separate deployment decisions — and the security implications are different for each.

Remember in Session 3 when we looked under the hood of the agent? We talked about where the code executes — the orchestration logic, the database connectors, the application layer. That code runs on your servers. You control it entirely. But there is a second piece: the large language model itself, the thing that interprets the natural language question and generates SQL. Where does that run? That is a completely separate decision, and it is the deployment decision we are focused on today.

Here is why this distinction matters. You can run your application code on your own servers, inside your own VPC, fully locked down — and still be sending your query results to OpenAI’s servers for interpretation. Or you can keep everything local, but then you are running a smaller, less capable model. These are different tradeoffs with different security implications. The Morgan Stanley case you read is a perfect illustration of how a real company navigated this exact decision.

Three Deployment Options

Cloud API

  • OpenAI, Anthropic, Google, etc.
  • Pay per query (~$0.01–0.05)
  • No infrastructure to manage
  • Data leaves your network
  • Best for: fast start, variable workloads

On-Premise / Private Cloud

  • Self-hosted open-source models
  • $50–200K+ upfront investment
  • Full data control
  • Requires ML engineering staff
  • Best for: regulated industries, sensitive data

Hybrid

  • Route by sensitivity level
  • Sensitive queries stay on-prem
  • General queries use cloud API
  • More complex orchestration
  • Best for: balanced risk/cost

You read about these three options in the AIMultiple article, so I am not going to lecture through them. Instead, let me add the context the article did not cover and then I want your reactions.

Cloud API is where almost everyone starts. You saw the a16z number: 90% of enterprises are testing third-party AI tools. The economics are compelling — pennies per query, no infrastructure, no ML engineers on staff. The catch is that your data leaves your network. For some of you, that is a non-starter. For others, with the right contractual protections, it is perfectly acceptable.

On-premise is the other end of the spectrum. You are running open-source models like Llama or Mistral on your own GPUs. Full data control, nothing leaves your building. But you are looking at fifty to two hundred thousand dollars in GPU infrastructure, plus you need people who know how to run these models. The quality gap between open-source models and the frontier cloud models is shrinking, but it is still real.

Hybrid is where most sophisticated enterprises are heading. You route queries by sensitivity level. Someone asks about public revenue data? Send it to the cloud API, get the best model. Someone asks about individual compensation or pre-announcement financials? Route that to the on-premise model. The orchestration is more complex, but you get the best of both worlds.

Cloud API dominates early adoption. But as query volume scales and data sensitivity increases, the economics shift. Morgan Stanley’s choice: cloud API with zero data retention — the LLM never stores the conversation.

Let me put some numbers on this. A simple cloud API query costs one to five cents. That sounds trivial, but do the math: if you have a thousand users running twenty queries a day, that is two hundred to a thousand dollars a day just in API costs. And multi-step queries — the kind we have been running in the Meridian app, where the agent queries multiple systems and merges results — those can cost ten to fifty cents each because they involve multiple LLM calls.

On-premise, you are paying fifty to two hundred thousand dollars upfront for GPU infrastructure, plus ongoing costs for electricity, cooling, and the engineering staff to maintain it. But after that, your marginal cost per query approaches zero.

The crossover point depends on volume and sensitivity. For a pilot with a hundred users, cloud API wins every time. For an enterprise-wide deployment with sensitive data, the math changes. Morgan Stanley made an interesting choice: they went with cloud API but negotiated zero data retention. The LLM processes the query and immediately forgets it. No conversation history is stored on the vendor’s servers. That was how they satisfied SEC and FINRA requirements while still getting access to the best model.

Four Data Security Approaches

Approach How It Works Tradeoff
Zero Data Retention LLM provider contractually deletes all inputs/outputs Simplest; still sends data to third party
Data Masking / Anonymization Strip PII before sending to LLM; re-identify on return Preserves analytical value; adds latency
On-Premise Models LLM runs inside your network; data never leaves Full control; highest cost and complexity
Role-Based Query Filtering Agent enforces permissions based on user identity Granular control; requires identity integration

Most production deployments combine two or more of these approaches. Morgan Stanley used zero data retention + role-based access.

You read about these four approaches in the data security primer. Let me add what the primer could not cover: how these approaches combine in practice. Almost no production deployment uses just one of these. Morgan Stanley used zero data retention plus role-based access. A healthcare company might use on-premise models plus data masking plus role-based filtering — three layers.

Let me run a quick poll. Think about your company and the use case you have been developing. Which combination of these approaches would your compliance team require? I want you to think about this concretely, because you are going to need it for the breakout exercise.

Here is what I see most often: companies in regulated industries — finance, healthcare, government — start with on-premise or zero data retention as the baseline, then layer role-based filtering on top. Companies in less regulated industries — tech, retail, manufacturing — often start with cloud API plus zero data retention and add role-based filtering as they scale.

The key insight is that these are not mutually exclusive. They are a menu, and your compliance team will want to see a combination that addresses data in transit, data at rest, and data access. Build vs. Buy

We have covered the “where does it run” question. Now the “who builds it” question. You may have read the DUNNIXER vendor scorecard with your Session 3 homework — if you did, you are ahead. If not, I will walk you through the framework now and we will apply it together.

[TIMING: Brief transition. Move to the next slide.]

The DUNNIXER Framework

Six Dimensions for Evaluating AI Solutions

  1. Data integration — How well does it connect to your existing systems?
  2. User experience — How intuitive is it for non-technical users?
  3. Nuance & accuracy — How well does it handle domain-specific language?
  4. Non-functional requirements — Security, compliance, scalability, uptime
  5. Implementation timeline — Weeks, months, or quarters to production?
  6. Xtensibility — Can you customize prompts, add tools, evolve it?

DUNNIXER gives you a structured way to evaluate any AI solution, whether you are looking at a vendor product or evaluating whether to build internally. Six dimensions, and every one of them matters.

Data integration is often the dealbreaker. If a vendor’s tool cannot connect to your Salesforce and your NetSuite and your legacy CRM, it does not matter how good the AI is. User experience determines adoption — if your sales team finds it easier to email the analyst than to use the AI tool, you have wasted your investment. Nuance and accuracy is about domain specificity: does it understand that “deal size” at your company means closed-won only? Non-functional requirements cover the boring but critical stuff: security certifications, uptime SLAs, compliance attestations. Implementation timeline is straightforward: can you get this live in weeks, or are you looking at a multi-quarter project? And extensibility is about the long game: can you customize the prompts, add new data sources, evolve the tool as your needs change, or are you locked into what the vendor ships?

Let us apply this framework together. We are going to score a specific vendor product — Salesforce Agentforce — on these dimensions. But first, let me show you the four strategic options you are choosing between.

[TIMING: 5 minutes. Walk through each dimension with a brief concrete example. Transition to the four strategic options slide.]

Four Strategic Options

Buy from Incumbent

  • Salesforce, Microsoft, SAP
  • Lowest risk, fastest start
  • Limited customization
  • You’re locked into their roadmap

AI-Native Startup

  • Purpose-built for AI-first workflows
  • Highest capability today
  • Vendor survival risk
  • May not integrate with legacy

Build Internally

  • Full control over every layer
  • Highest customization
  • Requires ML + engineering talent
  • Longest time to production

Hybrid / Extend

  • Buy a platform, customize on top
  • Example: Salesforce Agentforce
  • Moderate risk and control
  • Most common enterprise pattern

Four options, and most of you will land somewhere in the bottom right — the hybrid or extend approach. But let me walk through all four so you understand the tradeoffs.

Buying from your incumbent vendor — Salesforce, Microsoft, SAP — is the lowest risk path. They already have your data, they already have your security certifications, and their AI features are being bolted onto products you already use. The downside is limited customization. You get what they ship, on their timeline. If their AI copilot does not handle your specific workflow, you wait.

AI-native startups are purpose-built for AI-first workflows. They often have the highest capability today because AI is all they do. But there is vendor survival risk. Half of the AI startups that exist today will not exist in three years. And integration with your legacy systems may be nonexistent.

Building internally gives you full control. You saw in Session 3 that the core agent loop is fifty lines of code. But going from fifty lines to production is a different story — authentication, error handling, monitoring, audit logging. You need ML engineering talent, and you are looking at quarters, not weeks.

The hybrid or extend approach is the most common enterprise pattern. You buy a platform — Salesforce Agentforce, Microsoft Copilot Studio, something similar — and then you customize on top. You get the vendor’s infrastructure and security, but you add your own prompts, your own data connections, your own business logic.

Let me ask the room: where does your company typically land on build-versus-buy decisions for technology? Do you tend to buy from incumbents, or does your company have the engineering culture to build?

Example: Salesforce Agentforce

What it is: Salesforce’s AI agent platform — extends your existing CRM with conversational AI, pre-built connectors, and guardrails.

Strengths

  • Already integrated with your Salesforce data
  • Enterprise security and compliance built in
  • Low-code customization for business users
  • Salesforce handles model updates and scaling

Limitations

  • Only works well within the Salesforce ecosystem
  • Limited cross-system querying (NetSuite, Workday, etc.)
  • Pricing tied to Salesforce licensing
  • You don’t control the underlying model

Agentforce illustrates the extend strategy: buy the platform, customize the prompts and tools, accept the vendor’s guardrails.

Let us score this together using the DUNNIXER framework. I will take us through two or three dimensions as a group exercise.

Data integration: Agentforce gets a high score within the Salesforce ecosystem. If your CRM data is in Salesforce, the connection is native. But how does it score for cross-system queries? If you also need NetSuite financial data or Workday HR data, Agentforce’s reach drops significantly. Think about the Meridian app experience — we were querying across ten systems. Agentforce is strong within one.

Non-functional requirements: Salesforce already has SOC 2 certification, GDPR compliance, and enterprise-grade uptime SLAs. If your compliance team already trusts Salesforce with your data, the incremental trust required for Agentforce is small. That is a real advantage.

Extensibility: You can customize prompts and add tools through their low-code interface. But you do not control the underlying model. If Salesforce switches from one LLM provider to another, you go with them. That is the tradeoff of the extend strategy.

How would you score Agentforce on the remaining dimensions — user experience, nuance and accuracy, and implementation timeline? Anyone want to take a guess based on what you know?

Decision Matrix

Criterion Buy (Incumbent) AI-Native Startup Build Internally Hybrid / Extend
Time to value Weeks Weeks–months Quarters Months
Customization Low Medium High Medium–High

This matrix is your cheat sheet for the breakout exercise. Let me highlight the tensions.

Time to value versus customization: if you need something live in weeks, you are buying from an incumbent. But you are accepting low customization. If you need high customization, you are building internally, but you are looking at quarters to production.

Data control versus cost: full data control means building internally, which is the most expensive option. If you are willing to share data control with a vendor, you can cut costs significantly.

Risk is interesting because it comes in different flavors. Buying from an incumbent is lowest risk — they are not going anywhere, the product works, your compliance team already knows them. An AI-native startup carries vendor survival risk — will they exist in two years? Building internally carries execution risk — do you have the talent, and can you ship on time?

Look at the cost row. Year one costs range from thirty thousand dollars for a startup tool to over a million for an internal build. These are real numbers that your CFO will want to see. The hybrid approach typically lands in the hundred to three hundred thousand range, which is why it is the most common enterprise pattern.

You are going to use this matrix in the breakout. Keep it in mind.

What If You’re Not Ready?

Not every company is ready for an AI agent deployment. If you don’t have documented schemas, API access, or data catalogs, the honest answer may be: invest in data readiness first.

Signs You Should Wait

  • No one can describe your schema in plain English
  • API access requires a 6-month procurement process
  • Your data has no owner or steward
  • Compliance has never been asked about AI access to production data

The 60–90 days of pre-work from Session 2 applies here. Data readiness is the prerequisite that no vendor demo will mention.

I want to be honest with you about something that no vendor will tell you. Not every company is ready for this. If you are sitting there thinking “my company does not even have documented schemas” or “it would take six months just to get API access to our production databases,” then the honest answer for your capstone might be: the first phase of your AI strategy is data readiness, not AI deployment.

Remember Session 2, when we talked about the sixty to ninety days of pre-work? Cataloging your data, documenting your schemas, provisioning API access, classifying PII? That pre-work is not optional. It is the prerequisite. And if your company has not done it, that is actually a valuable finding for your capstone. Your deployment profile might say: “Phase 1 is data readiness, Phase 2 is a pilot deployment.”

There is no shame in that. In fact, a capstone that honestly says “we are not ready yet, and here is the six-month roadmap to get ready” is more credible than one that pretends you can deploy an AI agent next quarter when you cannot even describe your schema.
