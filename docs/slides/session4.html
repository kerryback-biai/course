How the Agent Loop Works

This opens the agent loop explanation (minutes 25-38). You are pulling back the curtain on the technical architecture. The goal is not to make students into developers, but to give them enough understanding to have informed conversations with technical teams and vendors.

The Core Loop

1

System Prompt

Schema description: what tables, columns, and relationships exist

2

Tool Call

LLM writes SQL and requests execution

3

Execute & Return

System runs SQL, sends results back to LLM

4

Test & Review

LLM checks results — correct? try again? deliver?

This is a test-and-review cycle, not a one-shot generation. The LLM writes code, the system executes it, the LLM reviews the results, and decides: deliver or try again.

Say: “Let me pull back the curtain. When you ask the Custom Chatbot a question, four things happen.”

Walk through each step: “Step one: the system prompt. Before your question ever reaches the LLM, a system prompt has been loaded that describes every table, every column, and every relationship. This is the institutional knowledge layer — it tells the LLM what data exists and what it means.”

“Step two: the tool call. The LLM reads your question, reads the schema, and writes SQL. But it does not just return the SQL. It requests execution — it says ‘run this query against this database.’”

“Step three: the system runs the SQL and sends the results back to the LLM. The LLM never touches the database directly.”

“Step four — and this is the key insight: the LLM reviews the results. If the result set is empty, or the numbers look wrong, or an error occurred, it tries again. It rewrites the SQL, changes a filter, fixes a join. This is a test-and-review cycle, not a one-shot generation.”

The Test-and-Review Cycle

What Makes This Different from a Dashboard

  • Dashboards run pre-written queries
  • If the query is wrong, you get wrong answers forever
  • Changing the question means changing the code
  • No self-correction capability

What the Agent Does

  • Writes a new query for every question
  • Reviews results: empty set? error? retry
  • Can rewrite SQL 2-3 times before delivering
  • Self-corrects — but not infallibly

Say: “This test-and-review cycle is what makes agents fundamentally different from dashboards. A dashboard runs pre-written queries. If the query has a bug, you get wrong answers until someone fixes the code. The agent writes a new query for every question, reviews the results, and can self-correct.”

“But — and this is important — self-correction is not infallible. The agent can retry 2 or 3 times and still get it wrong. It might think an empty result set means ‘no data’ when the real problem is a wrong filter. That is why the maker/checker framework matters. The agent corrects many of its own errors. You catch the rest.”

Deep Dive: Inside an Agent

This opens the deep-dive segment. Having built a data agent from English and seen the 50-line vs. 3,000-line gap, we now look inside a different agent — a portfolio analyst — to see exactly what those 50 lines do. The goal is not to turn students into developers, but to give them enough understanding to evaluate vendors, scope internal builds, and act as informed checkers.

What Is an Agent?

An agent is a program that:

  1. Maintains a conversation with a user
  2. Has access to tools (functions it can call)
  3. Decides which tools to use based on the question
  4. Chains multiple tool calls before responding

This is different from a script that always runs the same steps. The agent reasons about what information it needs and fetches it on the fly.

Say: “You just built an agent from English instructions. Now let me show you what is actually inside those 50 lines — using a different example that might hit closer to home.”

Why an Agent, Not a Script?

A Script

  • Fixed sequence of steps
  • Same analysis every time
  • Can’t handle follow-up questions
  • New analysis = new code

An Agent

  • Chooses tools based on context
  • Different path for each question
  • Conversational — remembers context
  • New analysis = just ask

You write the tools once; the agent figures out how to combine them.

Our Example: Portfolio Review Agent

We’ll walk through a portfolio analyst that can:

  • Look up current holdings and tax lots
  • Query a live price database
  • Compute returns, correlations, and statistics in Python
  • Compare sector weights against targets
  • Identify tax-loss harvesting opportunities
  • Find the most correlated Strong Buy replacement

The agent chooses its own path. Asking “review the portfolio” triggers different tools than “find tax-loss candidates in energy.”

Say: “This agent manages a stock portfolio. It has five tools — three return portfolio data, one runs SQL against a price database, and one runs Python for analytics. Think about your own portfolio: you might ask about sector allocation one day and tax-loss harvesting the next. The same five tools handle both questions — the agent decides which ones to call.”

Architecture

Five tools. Three return portfolio data. One executes SQL against a live price database. One runs Python for analytics.

Tools Are JSON Descriptions

Each tool has a name, description, and input_schema. The description tells the agent when and why to use it.

tools = [
    {
        "name": "get_holdings",
        "description": (
            "Retrieve all portfolio holdings with tax lot detail. "
            "Returns each position's ticker, sector, and individual "
            "lots (date acquired, shares, cost basis per share)."
        ),
        "input_schema": {
            "type": "object",
            "properties": {},
            "required": [],
        },
    },
    # ... more tools
]

The description is the critical part — it tells the agent when to use this tool. A tool with no inputs (like get_holdings) acts as a simple lookup.

Say: “Every tool the agent can use is defined by three things: a name, a description, and a schema of what inputs it accepts. The description is the most important part — it is how the agent decides when to call this tool. Think of it as a menu item: the name is the dish, the description is what is in it.”

The SQL and Python Tools

The agent can write its own queries and run its own analytics:

  - tickers: stock metadata

pandas, numpy, and json are

Multi-tool chains: run_sql fetches prices → run_python computes correlations. The agent decides the sequence.

The Tool Router

def execute_tool(name: str, inputs: dict) -> str:
    """Route a tool call to the appropriate handler."""
    if name == "get_holdings":
        return json.dumps(HOLDINGS)
    elif name == "get_target_allocation":
        return json.dumps(TARGET_ALLOCATION)
    elif name == "get_analyst_recommendations":
        sector = inputs["sector"]
        recs = STRONG_BUY.get(sector, [])
        return json.dumps({"sector": sector, "strong_buy": recs})
    elif name == "run_sql":
        return run_sql_query(inputs["sql"])
    elif name == "run_python":
        return run_python_code(inputs["code"])

This is your code — you control what each tool does. The agent only sees the tool’s name, description, and the result you return.

Say: “This is the traffic cop. When the agent says ‘I want to call get_holdings,’ this function catches that request and returns the actual data. When it says ‘run this SQL,’ the function connects to the database and runs it. You wrote the tools — you control what happens.”

Handling Tool Calls

if response.stop_reason == "tool_use":
    # Save the assistant's message
    messages.append({"role": "assistant", "content": response.content})

    # Execute each tool and collect results
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            result = execute_tool(block.name, block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result,
            })

    # Send results back to the agent
    messages.append({"role": "user", "content": tool_results})

The tool_use_id links each result to the corresponding call. The agent can make multiple tool calls in one turn.

The System Prompt

SYSTEM_PROMPT = """\
You are a portfolio analyst assistant. You help review a client
portfolio, identify tax-loss harvesting opportunities, and
recommend replacement securities.

When harvesting losses, sell the lots with the largest unrealized
loss first (highest cost basis relative to current price).

When recommending a replacement, compute return correlations
between the stock being sold and each Strong Buy candidate,
then recommend the most highly correlated candidate.

When writing SQL, use closeadj for return calculations.
"""

The system prompt gives the agent its strategy — domain expertise that guides tool selection. Compare this to the XYZ Corp system prompt you saw earlier — same pattern, different domain.

Example: Tax-Loss Harvesting

You: Harvest the INTC losses. What should I buy to replace the exposure?

The agent:

  1. Identifies INTC lots with the largest unrealized losses
  2. Calculates the realized loss
  3. Checks if selling moves Technology below target weight
  4. Calls get_analyst_recommendations("Technology")
  5. Calls run_sql to fetch daily prices for INTC and each candidate
  6. Calls run_python to compute return correlations
  7. Recommends the most highly correlated Strong Buy stock

Seven tool calls across three tools, none predetermined. The path depends on what the human asks and what the data shows.

Say: “One question. Seven tool calls. Three different tools. And none of that was predetermined — the agent decided the sequence based on the question and what it found along the way. This is the same loop you just saw, running seven times before it had enough information to answer.”
