How the Agent Loop Works

This opens the agent loop explanation (minutes 25-38). You are pulling back the curtain on the technical architecture. The goal is not to make students into developers, but to give them enough understanding to have informed conversations with technical teams and vendors.

The Core Loop

1

System Prompt

Schema description: what tables, columns, and relationships exist

2

Tool Call

LLM writes SQL and requests execution

3

Execute & Return

System runs SQL, sends results back to LLM

4

Test & Review

LLM checks results — correct? try again? deliver?

This is a test-and-review cycle, not a one-shot generation. The LLM writes code, the system executes it, the LLM reviews the results, and decides: deliver or try again.

Say: “Let me pull back the curtain. When you ask the Data Simulation a question, four things happen.”

Walk through each step: “Step one: the system prompt. Before your question ever reaches the LLM, a system prompt has been loaded that describes every table, every column, and every relationship. This is the institutional knowledge layer — it tells the LLM what data exists and what it means.”

“Step two: the tool call. The LLM reads your question, reads the schema, and writes SQL. But it does not just return the SQL. It requests execution — it says ‘run this query against this database.’”

“Step three: the system runs the SQL and sends the results back to the LLM. The LLM never touches the database directly.”

“Step four — and this is the key insight: the LLM reviews the results. If the result set is empty, or the numbers look wrong, or an error occurred, it tries again. It rewrites the SQL, changes a filter, fixes a join. This is a test-and-review cycle, not a one-shot generation.”

The System Prompt Is Everything

What the LLM sees before your question

You are a data analyst for XYZ Corp, a $500M B2B distributor.

Available databases:
- sf_opportunities: Salesforce CRM (Industrial division)
  Columns: OpportunityId, AccountName, Amount, CloseDate, Stage...
- legacy_orders: Legacy CRM (Energy division)
  Columns: order_id, customer_name, order_amount, order_date (VARCHAR!)...
- hs_deals: HubSpot CRM (Safety division)
  Columns: deal_id, company_name, amount, close_date...
- wd_employees: Workday HR
  Columns: employee_id, division, department, hire_date, salary...

IMPORTANT: legacy_orders stores dates as MM/DD/YYYY text strings.
IMPORTANT: Customer names differ across systems. Use fuzzy matching.

Say: “This is a simplified version of what the LLM sees before your question arrives. The system prompt describes every table, every column, and every gotcha. Look at the notes at the bottom: the Legacy CRM stores dates as text strings, not proper dates. Customer names differ across systems — use fuzzy matching.”

“This system prompt is the institutional knowledge layer. If the agent gets a query wrong, the fix is often not in the code — it is in the schema description. Change the description, change the SQL.”

“Who writes this? In a production deployment, it is a combination of data engineers and business analysts. Someone who knows both the technical schema and what the business actually means when it says ‘revenue’ or ‘active customer.’”

The Test-and-Review Cycle

What Makes This Different from a Dashboard

  • Dashboards run pre-written queries
  • If the query is wrong, you get wrong answers forever
  • Changing the question means changing the code
  • No self-correction capability

What the Agent Does

  • Writes a new query for every question
  • Reviews results: empty set? error? retry
  • Can rewrite SQL 2-3 times before delivering
  • Self-corrects — but not infallibly

Say: “This test-and-review cycle is what makes agents fundamentally different from dashboards. A dashboard runs pre-written queries. If the query has a bug, you get wrong answers until someone fixes the code. The agent writes a new query for every question, reviews the results, and can self-correct.”

“But — and this is important — self-correction is not infallible. The agent can retry 2 or 3 times and still get it wrong. It might think an empty result set means ‘no data’ when the real problem is a wrong filter. That is why the maker/checker framework matters. The agent corrects many of its own errors. You catch the rest.”

From Ad Hoc to Automated

The Production Pattern

What You Have Been Doing

  • Type a question in a chat interface
  • Get an answer in real time

  • You still have to remember to ask

What Production Looks Like

  • Same agent loop, triggered by a schedule
  • Monday morning dashboard refresh
  • Monthly board deck, weekly sales forecast
  • Error handling + version control + human review gates

Say: “Everything you have done so far has been interactive. You type a question, you get an answer. That is useful. But in production, these pipelines run on schedules.”

“Imagine the quarterly executive summary you just saw, but it runs automatically on the first Monday of every quarter. It queries the same six systems, generates the same charts, writes the same narrative. But instead of you typing the prompt, a cron job triggers it.”

“What changes in production? Three things: error handling — what happens when a database is down or a query returns unexpected results? Version control — can you reproduce last month’s report exactly? And human review gates — the report is drafted but held for approval before distribution.”

Iteration Tips

Getting Better Output

  • Be specific about format: “Traffic-light dashboard” vs. “one-page executive brief”

Where Does the Code Run?

Brief preview segment (minutes 80-85). This sets up Session 5 on deployment architecture. Keep it high-level — 5 minutes, not a deep dive.

Three Execution Options

Cloud LLM Sandbox

  • Code runs on the vendor’s servers
  • Easiest to set up
  • Your data leaves your network
  • What we use for the XYZ Corp demo (synthetic data)

Your Own Containers

  • Code runs on your servers
  • Data stays on-premise
  • You manage infrastructure
  • Higher control, higher cost

SQL-Only Mode

  • LLM only generates SQL, never sees raw data
  • Your system executes the SQL locally
  • Minimum data exposure
  • Most restrictive but most secure

Our demo uses synthetic data in the cloud. Production deployments keep data on-premise. Session 5 digs into these tradeoffs in detail.

Say: “Quick preview. When you deploy an agent in production, where does the code actually run? Three options.”

Walk through briefly: “Option one: the vendor’s cloud. Easiest to set up, but your data leaves your network. That is what we use for the XYZ Corp demo — the data is synthetic, so it does not matter. Option two: your own containers. Data stays on-premise. You manage everything. Higher control, higher cost. Option three: SQL-only mode. The LLM generates SQL but never sees the raw data. Your system executes the query locally and only sends aggregate results to the LLM.”

“Each option has different security, cost, and capability tradeoffs. Session 5 is entirely about this decision. The pre-readings I am assigning tonight will prepare you for that discussion.”
