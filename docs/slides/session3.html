The Agent Loop

This section divider opens the agent loop demo segment, which runs from roughly minute 10 to minute 30. Say: “The Agent Loop. This is the core pattern.” Then advance.

Before you advance, make sure the Meridian app is open in another window and ready for the demo. You will be showing a cross-system question and walking through what happens behind the scenes — the system prompt, the tool call, the SQL execution, the merge, and the final response.

What Happens When You Ask a Question?

1

System Prompt

Schema description tells the LLM what tables and columns exist

2

Tool Call

LLM writes SQL (or Python) and requests execution

3

Execute & Return

System runs the code and sends results back to the LLM

4

Review & Deliver

LLM checks results, fixes errors or delivers the answer

This is the first conceptual slide of the agent loop segment. This segment should take about 20 minutes total (minutes 10-30), including the live demo. Walk through each step deliberately, pointing to each card as you go.

Say: “When you type a question into the Meridian app — or any AI data agent — four things happen. Step one: the system prompt. Before your question even reaches the LLM, a system prompt has already been loaded that describes the database schema — what tables exist, what columns they have, what the relationships are. Think of this as giving the LLM a map of your data before asking it to navigate.”

“Step two: the tool call. The LLM reads your question, reads the schema, and writes SQL — or sometimes Python — to answer it. But here is the key: it does not just return the SQL to you. It makes a tool call, which is a request to the system to actually execute that code.”

“Step three: execute and return. The orchestration layer — the code that sits between you and the LLM — takes that SQL, runs it against the database, and sends the results back to the LLM. The LLM never touches the database directly. It writes code, and the system executes it.”

“Step four: review and deliver. The LLM looks at the results. If something looks wrong — an empty result set, an error message, numbers that do not make sense — it can try again. It might rewrite the SQL, change a filter, fix a join. If the results look good, it formats them into a natural language response with charts and delivers the answer.”

Transition: “Let me show you this in action. I am going to switch to the Meridian app and ask a cross-system question, but this time I want you to watch what happens behind the scenes.”

The Key Insight: Test and Review

The agent doesn’t just generate code and hope for the best. It writes code, executes it, reviews the result, and decides whether to deliver or try again.

This is the same pattern a careful analyst follows: write a query, check the output, refine if something looks off, then present the final answer. The LLM automates that entire cycle.

After the demo, come back to this slide to reinforce the key insight. Say: “Here is what I want you to take away from what you just saw. The agent does not just generate code and hope for the best. It writes code, executes it, reviews the result, and decides whether to deliver or try again. That loop — write, execute, review, decide — is the fundamental pattern.”

“Think about how a careful analyst works. They write a query, look at the output, notice something is off, rewrite the query, check again, and only then present the results. The LLM automates that entire cycle. It is not magic — it is a structured test-and-review process.”

“This is also why the agent sometimes takes a few seconds longer than you might expect. It is not slow — it is reviewing its own work. And sometimes it runs through the loop two or three times before delivering an answer, because the first attempt did not look right.”

Ask the room: “Did anyone notice during Sessions 1 or 2 that the agent sometimes seemed to pause or take a moment before answering? That was the review step. It was checking its own work.”

The Agent Loop in Pseudocode

agent_loop.py — The Core (~50 Lines)

def agent_loop(question, schema):
    messages = [system_prompt(schema), user_message(question)]

    for attempt in range(3):
        response = llm.chat(messages)
        if response.has_tool_call:
            result = execute(response.tool)
            messages.append(result)        # feed back
        else:
            return response.text           # done

This slide shows the pseudocode for the core agent loop. Walk through it line by line. Say: “Here is the agent loop in pseudocode. This is a simplified version of what the Meridian app runs at its core. Let me walk through it.”

“First, we set up the messages list. It starts with two things: the system prompt, which contains the schema description, and the user’s question. That is all the LLM needs to get started.”

“Then we enter a loop — up to three attempts. On each pass, we send the messages to the LLM and get a response. If the response contains a tool call — meaning the LLM wants to run some SQL or Python — we execute it, take the result, and append it to the messages list. Now the LLM can see what happened. On the next iteration, it will either make another tool call or, if it is satisfied with the results, return a text response.”

“If the response does not contain a tool call, we are done — the LLM is delivering its final answer.”

“That is it. That is the core. The real code has error handling, logging, and some other details, but the logic is this simple. About 50 lines of Python.”

The System Prompt Is the Secret

This section divider opens the system prompt segment, which runs from roughly minute 30 to minute 45. Say: “The System Prompt Is the Secret.” Then advance.

This is a critical segment because it connects the technical architecture to something executives can directly influence. The system prompt is where business knowledge meets technology, and getting it right is the difference between an agent that works and one that does not.

Why the Prompt Matters

Without Join Key Documentation

  • LLM sees orders and customers tables
  • No information about shared keys
  • Agent can’t figure out how to merge
  • Query fails or returns wrong results

With Join Key Documentation

  • Prompt says: orders.customer_id links to customers.id
  • Agent writes correct JOIN
  • Merges work on first attempt
  • Accurate, complete answers

This slide sets up the live demo of prompt manipulation. This segment should take about 15 minutes total (minutes 30-45). Say: “The system prompt is the most important piece of the entire system. Let me show you why with a simple example.”

“On the left: without join key documentation. The LLM can see that there are two tables — orders and customers — but it has no information about how they relate. It does not know that orders.customer_id links to customers.id. Without that, it cannot write a correct JOIN, and either the query fails or it returns garbage.”

“On the right: with join key documentation. We add one line to the system prompt — ‘orders.customer_id links to customers.id’ — and suddenly the agent writes correct JOINs on the first attempt. Merges work. Answers are accurate.”

“This is not a hypothetical. Let me show you.”

The Prompt Is “Institutional Knowledge”

The system prompt encodes everything the agent needs to know about your data: table names, column meanings, relationships, business rules, and formatting preferences.

Change the prompt, change the behavior. Add a rule like “always filter to active customers” and every query respects it. Remove join documentation and merges break. The prompt is the most important piece of the system.

Say: “The system prompt is institutional knowledge. It is everything your agent needs to know about your data, encoded in plain English. Table names, column meanings, relationships between tables, business rules, formatting preferences — all of it lives in the prompt.”

“And here is the powerful implication: change the prompt, change the behavior. Add a rule that says ‘always filter to active customers’ and every query the agent writes will respect that rule. Add a formatting preference that says ‘always show dollar amounts with commas and two decimal places’ and every response will follow it. Remove the join documentation and merges break.”

“This means that the people who write and maintain the system prompt are effectively the people who control the agent’s behavior. As your data changes, as business rules evolve, the prompt needs to evolve with it.”

Where Does the Code Run?

This section divider opens the execution architecture segment, which runs from roughly minute 45 to minute 60. Say: “Where Does the Code Run?” Then advance.

This is a lecture segment, not a demo. The key point is that the LLM generates code but something else has to execute it, and that “something else” is an architectural decision with security implications. This is the first of several deployment decisions executives need to understand — it sets the stage for Session 5.

Three Architecture Options

A. Cloud LLM Sandbox

  • Vendor runs the code (OpenAI, Anthropic)
  • Simplest to set up
  • Data leaves your network
  • Fine for non-sensitive data

B. Your Own Containers

  • Docker on your servers or cloud
  • Production answer for sensitive data
  • You control the execution environment
  • More infrastructure to manage

C. SQL-Only Mode

  • No Python execution at all
  • LLM generates SQL, your DB runs it
  • Safest for regulated environments
  • Can’t do in-memory merges or charts

Walk through each option. Say: “The LLM generates SQL and Python. Something has to execute it. There are three main options, and each involves a different tradeoff.”

“Option A: Cloud LLM Sandbox. This is the simplest. The vendor — OpenAI, Anthropic, whoever — runs the code on their servers. You send the query, they execute it, they send back the results. The advantage is simplicity: nothing to set up, nothing to maintain. The disadvantage is that your data leaves your network. The query results pass through the vendor’s servers. For non-sensitive data or for prototyping, this is fine. That is what our Meridian demo uses — synthetic data running through a cloud API.”

“Option B: Your Own Containers. This is the production answer for sensitive data. You run Docker containers on your own servers or in your own cloud environment. The LLM still generates the code, but the execution happens inside your infrastructure. Data never leaves your network. The tradeoff is more infrastructure to manage — you need container orchestration, monitoring, security patching.”

“Option C: SQL-Only Mode. The LLM generates SQL queries, but it never runs Python at all. The SQL goes directly to your database, and the database executes it. This is the safest option for highly regulated environments because the only code that runs is SQL against your own database. The tradeoff is that you lose the ability to do in-memory merges, data transformations, and chart generation. Remember those cross-system merges from Session 2? Those required Python. In SQL-only mode, you cannot do them.”

Ask the room: “Which of these three options do you think would fit your organization? Think about your data sensitivity, your IT infrastructure, and your compliance requirements.” Let a few people share quick thoughts, but do not let this become a long discussion — save the deeper deployment conversation for Session 5.

Choosing the Right Option

Cloud Sandbox Your Containers SQL-Only
Setup effort Low Medium Low
Data stays on-prem No Yes Yes
Python/charting Yes Yes No
Best for Prototyping Production Regulated

Say: “Here is the summary table. Cloud sandbox is low effort but data leaves your network. Your own containers keep data on-prem and support Python, but require more setup. SQL-only is low effort and keeps data on-prem, but you lose charting and in-memory processing.”

“For most organizations, the answer ends up being Option B for production — your own containers. You prototype with Option A, prove the concept, and then move to containers when you are ready to connect to real data. Some regulated industries — banking, healthcare — might start and stay with Option C.”

“We will revisit this in much more depth in Session 5, when we talk about deployment architecture and data security. For now, the important thing is that you understand this is a decision that needs to be made, and it has real implications for what the agent can and cannot do.”
