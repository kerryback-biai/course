Two Types of Prediction

Let us start with the most fundamental distinction in predictive modeling. Every business prediction falls into one of two categories, and knowing which one you are dealing with determines everything that follows — the model you build, the metrics you use, and how you interpret the results.

[TIMING: 15 seconds. Section divider — advance immediately.]

Classification vs. Regression

Classification: “Which category?”

  • Will this customer churn? Yes / No
  • Is this transaction fraudulent? Fraud / Legitimate
  • Which segment does this lead belong to? Hot / Warm / Cold
  • Will this employee leave within 12 months?
  • Will this loan default?

Regression: “How much?”

  • How much will this customer spend next quarter? $12,400
  • What will demand be in March? 4,200 units
  • What price maximizes margin? $89.99
  • How many support tickets will we get next week?
  • What will Q3 revenue be?

Every business prediction is one of these two types. The type determines how you build the model and how you evaluate it.

Here is the fundamental split. Classification predicts categories — yes or no, fraud or legitimate, hot or warm or cold. Regression predicts continuous numbers — dollar amounts, quantities, prices.

The distinction matters because the evaluation is completely different. For classification, you care about things like: how often does the model say “churn” when the customer actually stays? That is a false positive, and it has a business cost — you might offer an unnecessary discount. For regression, you care about how far off the predictions are on average.

How Gradient Boosting Works

🌳

Tree 1

Build a small decision tree. It gets some predictions right, some wrong.

🎯

Find Errors

Look at what Tree 1 got wrong. Focus on those mistakes.

🌳🌳

Tree 2

Build a new tree that specifically corrects Tree 1’s errors.

🔄

Repeat

Keep adding trees. Each one fixes the remaining errors. Stop when improvement plateaus.

The key insight: No single tree is smart. But hundreds of trees, each correcting the previous one’s mistakes, produce remarkably accurate predictions. This is ensemble learning — the wisdom of many weak learners.

Here is the conceptual explanation. Gradient boosting builds many small decision trees sequentially. The first tree makes predictions — some right, some wrong. The second tree does not start from scratch; it focuses specifically on the cases the first tree got wrong. The third tree corrects the remaining errors from tree two. And so on, for hundreds or thousands of trees.

No single tree is particularly smart. Each one is a “weak learner” — it might be only slightly better than random guessing. But the combination of hundreds of weak learners, each correcting the previous one’s mistakes, produces a model that is remarkably accurate. This is called ensemble learning.

Think of it like editing a document. The first draft has errors. A second pass catches some of those errors but might miss others. A third pass catches what the second missed. After enough passes, the document is polished. That is gradient boosting — many passes, each one cleaning up what the previous pass missed.

You do not need to remember any of this to use the technique. Claude will handle the implementation. But when a vendor tells you “we built a machine learning model for churn prediction,” you now know enough to ask: “What algorithm? Gradient boosting? How many trees? What was the cross-validation score?” Those questions signal that you are not someone who can be sold a black box.

What Is a Decision Tree?

A decision tree asks a series of yes/no questions about your data to reach a prediction:

Example: Will this customer churn?

Is monthly usage < 5 sessions? ├── YES → Has tenure < 12 months? │ ├── YES → CHURN (85% confidence) │ └── NO → Has open support ticket? │ ├── YES → CHURN (70% confidence) │ └── NO → STAY (60% confidence) └── NO → Has usage declined 3 months in a row? ├── YES → CHURN (65% confidence) └── NO → STAY (90% confidence)

Each tree in gradient boosting is small and simple — maybe 4-8 questions deep. The power comes from combining hundreds of these simple trees.

Before we move on, let me make decision trees concrete. A decision tree is just a series of yes/no questions. Is monthly usage less than five sessions? If yes, go left. Has the customer been with us less than a year? If yes, predict churn with eighty-five percent confidence.

Each individual tree in a gradient boosting model is deliberately kept small — maybe four to eight questions deep. This is by design. A deep, complex tree would memorize the training data instead of learning generalizable patterns. By keeping each tree simple and combining hundreds of them, gradient boosting gets the best of both worlds: complexity in aggregate, simplicity in each component.

You will see this in action in a few minutes when we build a model with Claude.

The ML Workflow

Now let us walk through the end-to-end workflow for building a predictive model. This is the sequence you will follow in the hands-on exercise, and it is the same sequence that data scientists follow in production. The difference is that you will have Claude write the code while you focus on evaluating the output.

[TIMING: 15 seconds. Section divider — advance immediately.]

Five Steps to a Predictive Model

📊

Prepare Data

Clean, explore, handle missing values

✂️

Split

Train set (80%) and test set (20%)

⚙️

Fit Model

Train gradient boosting on the training set

🔮

Predict

Apply model to the test set it has never seen

✅

Evaluate

Confusion matrix, accuracy, feature importance

Why split the data? If you evaluate a model on the same data it trained on, it is like grading a student on questions they already saw. The test set is the final exam with new questions.

Here is the five-step workflow. Step one: prepare the data — explore it, handle missing values, understand what each column means. Step two: split the data into a training set and a test set. The model learns from the training set and gets evaluated on the test set — data it has never seen. Step three: fit the model — this is where gradient boosting builds its hundreds of trees. Step four: predict — apply the trained model to the test set. Step five: evaluate — look at the confusion matrix, accuracy metrics, and feature importance.

The most important concept here is the train/test split. Why do we split the data? Because if you evaluate a model on the same data it trained on, you are measuring memorization, not generalization. It is like giving a student the final exam, then grading them on the exact same questions. Of course they will score well — they have seen the answers. The test set is a final exam with new questions. If the model performs well on data it has never seen, you can trust that it has learned real patterns, not just memorized the training data.

Cross-validation takes this further. Instead of one split, it does five or ten different splits and averages the results. This gives you a more reliable estimate of how the model will perform on truly new data. Claude will handle cross-validation automatically when you ask for it.

Evaluating Models

Now that you have seen what a model produces, let us talk about how to evaluate it. This is where the maker/checker framework applies most directly. The AI built the model — now you need to decide whether to trust it.

[TIMING: 15 seconds. Section divider — advance immediately.]

Reading a Confusion Matrix

What the Matrix Tells You

  • True Positives (272): Correctly predicted churn — these are your wins
  • True Negatives (896): Correctly predicted stay — no action needed
  • False Positives (139): Predicted churn, but customer stayed — unnecessary retention spend
  • False Negatives (102): Predicted stay, but customer churned — missed opportunity, the costly error

The Business Question

  • What is the cost of each error type?
  • False positive: you offer a discount to someone who was staying anyway (~$50/customer?)
  • False negative: you lose a customer worth $1,200/year in revenue
  • False negatives are 24x more costly — optimize for recall

The confusion matrix is not just a statistics table — it is a business decision tool. Let me show you why.

We have two types of errors. False positives: 139 customers the model flagged as likely to churn, but they were actually going to stay. What is the cost? You might offer them a retention discount unnecessarily — say fifty dollars each. That is about seven thousand dollars.

False negatives: 102 customers the model predicted would stay, but they actually churned. What is the cost? Each lost customer represents maybe twelve hundred dollars per year in revenue. That is a hundred and twenty-two thousand dollars in lost revenue.

False negatives are twenty-four times more costly than false positives in this example. That changes how you evaluate the model. You do not want the model with the highest accuracy — you want the model that catches the most churners, even if it means more false alarms. This is the precision-recall tradeoff, and it is a business decision, not a statistical one.

When you evaluate a model, the first question is not “what is the accuracy?” It is “what is the cost of each type of error?” That determines which metric matters most.

[TIMING: 4 minutes. Move to precision vs recall.]

Precision vs. Recall: A Business Tradeoff

Precision (66.2%)

  • “When we predict churn, how often are we right?”
  • 272 correct out of 411 churn predictions
  • Higher precision = fewer wasted retention offers
  • Optimize for precision when false positives are expensive
  • Example: fraud alerts that freeze customer accounts

Recall (72.7%)

  • “Of all actual churners, how many did we catch?”
  • 272 caught out of 374 actual churners
  • Higher recall = fewer missed churners
  • Optimize for recall when false negatives are expensive
  • Example: cancer screening, churn prevention, fraud in high-value accounts

You cannot maximize both. Catching more churners (higher recall) means more false alarms (lower precision). The right balance depends on your business costs.

Precision and recall are inversely related — you cannot maximize both simultaneously. Increasing recall means casting a wider net, which catches more true churners but also flags more customers who were not going to churn. Increasing precision means being more selective, which reduces false alarms but misses more actual churners.

The right tradeoff depends entirely on your business context. For churn prevention, where a lost customer is worth far more than a wasted discount, you optimize for recall. For fraud detection where blocking a legitimate transaction angers a customer, you might balance more toward precision. For cancer screening, you overwhelmingly optimize for recall — you never want to miss a true positive.

This is a judgment call that AI cannot make for you. The model can give you any tradeoff you want by adjusting a threshold. Your job is deciding which tradeoff is right for the business.

Regression Metrics: R² and RMSE

R² (R-squared)

  • “How much of the variation does the model explain?”
  • R² = 0.85 means the model explains 85% of variation in the target
  • R² = 1.0 is a perfect model (suspicious — probably data leakage)
  • R² = 0.0 means the model is no better than predicting the average
  • Business rule of thumb: R² > 0.7 is usually useful for planning

RMSE (Root Mean Squared Error)

  • “On average, how far off are the predictions?”
  • RMSE = $4,200 means predictions are off by ~$4,200 on average
  • In the same units as the target variable (dollars, units, etc.)
  • Ask: “Is this error margin acceptable for the business decision?”
  • RMSE of $4,200 on $50K deals is fine; on $8K deals, it is not

For regression problems — predicting continuous numbers — you use different metrics. R-squared tells you how much of the variation in the target your model explains. An R-squared of 0.85 means the model captures eighty-five percent of the variation in, say, quarterly revenue. The remaining fifteen percent is noise the model cannot explain.

RMSE — root mean squared error — tells you how far off the predictions are on average, in the same units as your target. If you are predicting deal values and the RMSE is four thousand two hundred dollars, that means the model’s predictions are off by about four thousand two hundred dollars on average.

The business question for RMSE is always: is this error margin acceptable for the decision I am making? An RMSE of four thousand two hundred dollars on fifty-thousand-dollar deals is fine for forecasting. The same RMSE on eight-thousand-dollar deals means the model is off by more than fifty percent on average — probably not useful.

You will not build a regression model in today’s exercise, but you will encounter these metrics when you or your team work on forecasting and demand planning.

The Maker/Checker Framework for Models

  1. Better Than Baseline?
  • Does the model beat a naive approach?
  • Baseline for churn: predict “stay” for everyone → 73.5% accuracy
  • Our model: 82.9% accuracy
  • 9.4 percentage points better — real improvement
  1. Features Make Sense?
  • Do the top predictors match domain knowledge?
  • Contract type, tenure, charges → makes sense
  • Customer ID, row number → data leakage
  • Ask a domain expert: “Do these surprise you?”
  1. Generalizes Well?
  • Cross-validation: 81.4% ± 1.8%
  • Training accuracy: 85.2%
  • Small gap → model is generalizing
  • If train = 99% but CV = 60% → overfitting

Here is your three-question checklist for evaluating any predictive model. You do not need to understand the math. You need to ask these three questions.

Question one: does the model beat a baseline? The simplest baseline for churn is “predict everyone stays” — that gives you 73.5 percent accuracy for free. Our model gets 82.9 percent. That is a real improvement of 9.4 percentage points. If a model only beats the baseline by one or two points, it may not be worth the complexity.

Question two: do the important features make business sense? We already covered this. Contract type, tenure, and monthly charges as top predictors? That matches what any experienced manager would tell you. If the top features do not make sense, the model may be exploiting a data artifact.

Question three: does the model generalize? Compare the cross-validation score to the training score. Our model gets 85.2 percent on training data and 81.4 percent on cross-validation. That is a small gap, which means the model learned real patterns, not memorized the training data. If you saw training accuracy of 99 percent but cross-validation accuracy of 60 percent, that model is memorizing, not learning. It will fail on new data.

Three questions. That is your entire evaluation framework. AI builds the model — you ask these three questions.

Red Flags: When Not to Trust the Model

🚩 Too Good to Be True

  • 99% accuracy usually means data leakage
  • The target variable leaked into the features
  • Example: including “cancellation_date” to predict churn

🚩 Nonsense Features

  • Customer ID, row number, or timestamp as top features
  • The model is memorizing, not learning
  • Always check feature importance before trusting predictions

🚩 Big Train/Test Gap

  • Train accuracy: 98%
  • Test accuracy: 65%
  • The model memorized training data and fails on new data

🚩 Unstable Cross-Val

  • Scores: 85%, 62%, 78%, 91%, 55%
  • High variance across folds
  • Model is sensitive to which data it sees — not reliable

Four red flags that should make you stop and investigate before deploying a model.

First: too good to be true. If you see 99 percent accuracy, do not celebrate — investigate. Almost always, this means data leakage: information about the outcome leaked into the input features. The classic example is including “cancellation_date” as a feature when predicting churn. Of course the model is perfect — it can see the answer in the input.

Second: nonsense features at the top of the importance list. Customer ID, row number, account creation timestamp — these should never be meaningful predictors. If they are, the model is memorizing specific customers rather than learning generalizable patterns.

Third: a large gap between training and test performance. If the model gets 98 percent on training data but 65 percent on test data, it has memorized the training set. It learned the noise, not the signal.

Fourth: unstable cross-validation scores. If the model gets 85 percent on one fold and 55 percent on another, it is highly sensitive to which data it sees. That is not a model you can rely on for business decisions.

Any of these red flags means you need to go back and investigate before trusting the predictions.

Prompt: Challenge the Model

PROMPT: What would a skeptical VP of Customer Success push back on if I presented this model? What are the weaknesses?

Claude’s Response (simulated)

A skeptical VP would likely raise these concerns:

  1. “We’re missing 27% of churners.” The recall of 72.7% means 102 customers who churned were not flagged. If each lost customer costs $1,200/year, that’s $122K in missed revenue.

  2. “Contract type is the top feature — we already know that.” The model confirms what the team already suspects. The VP wants to know: what does the model tell us that we DON’T already know? → Look at features 4-10 for non-obvious insights.

  3. “This is trained on historical data.” If we just changed our pricing or launched a new product, past churn patterns may not predict future behavior. The model needs retraining regularly.

  4. “What do we DO with the predictions?” A model is only useful if it drives action. What’s the intervention for flagged customers? Who owns the follow-up?

This is what step five looks like in practice. When you ask Claude to play the role of a skeptical VP, it surfaces exactly the right concerns. The recall gap means we are missing a quarter of churners. The top features confirm existing intuition rather than revealing new insights. The model is trained on historical data that may not reflect current conditions. And most importantly: what do you actually do with the predictions?

That fourth point is the most important. A model that predicts churn is only valuable if someone acts on the predictions. Who gets the list of at-risk customers? What is the intervention? Three Things to Remember

AI Builds the Model

  • You do not need to write code
  • You do not need to understand the math
  • You need to write clear prompts and evaluate the output

Three Questions to Ask

  • Better than baseline?
  • Features make business sense?
  • Generalizes to new data?
  • If yes to all three, proceed with caution

Predictions Need Action

  • A model without action is just a report
  • Who receives predictions?
  • What decision changes?
  • This is what separates strategy from spectacle

Three takeaways. First: AI builds the model, you evaluate it. You did this today — you prompted Claude, it wrote the code, and you evaluated the confusion matrix, feature importance, and cross-validation scores. That is the workflow.

Second: three questions. Does it beat the baseline? Do the features make sense? Does it generalize? If the answer to all three is yes, you have a model worth exploring further. If any answer is no, investigate before trusting the predictions.

Third: predictions need action. This is the most important takeaway. A churn model is only valuable if someone calls the at-risk customers. A demand forecast is only valuable if someone adjusts inventory. Your capstone should specify not just what AI does, but what humans do differently because of it.
